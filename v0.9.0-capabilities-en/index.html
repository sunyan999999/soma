    Version Overview¶

    SOMA v0.9.0 is the largest architectural upgrade since v0.1.0. While v0.8.0 completed the individual agent's cognitive reasoning loop (causal reasoning, conflict detection, cross-domain analogy, quality evaluation), v0.9.0 achieves the leap from individual to collective intelligence — enabling multiple SOMA Agents to collaborate as a team, each with independent memory, independent evolution paths, and specialized expertise domains, forming collective intelligence that surpasses any individual through expert routing, distributed evolution, and consensus protocols.

    Evolution Roadmap¶

    v0.5.0 — Foundation: 7 Thinking Laws + Vector Retrieval + Evolution Loop
v0.6.0 — Reasoning Templates: Structured Reasoning Framework + Benchmark System
v0.7.0 — Memory Intelligence: Consolidation + Active Forgetting + External Knowledge Import
v0.8.0 — Cognitive Reasoning: Causal Chains + Conflict Detection + Cross-Domain Analogy + Quality Evaluation
v0.9.0 — Collective Intelligence: Multi-Agent Collaboration + Expert Routing + Distributed Evolution + Consensus ← Current

    Five Core Capabilities¶

    1. AgentRegistry¶

    Formalizing Domain Expertise

    Each Agent is no longer a homogeneous "general thinker" — instead, each is a specialized intelligent entity with explicit expertise domain labels, success rate tracking, and session statistics.

    from soma.multi_agent import AgentRegistry, AgentInfo

registry = AgentRegistry()

# Register a legal expert agent
registry.register(
    agent=law_agent,
    expertise=["legal", "contracts", "litigation", "intellectual property"],
    description="Specializes in legal text analysis, contract review, and compliance assessment",
    is_default=False,
)

# Register a technical architect agent
registry.register(
    agent=tech_agent,
    expertise=["technology", "architecture", "performance optimization", "system design"],
    description="Specializes in distributed systems architecture, performance bottleneck analysis, and technology selection",
)

# Find experts by domain
experts = registry.find_experts("legal")
# → [(law_agent, 1.0), ...]

# Get all agent statistics
for info in registry.list_agents():
    print(f"{info.agent_id}: {info.expertise} "
          f"(sessions={info.session_count}, sr={info.success_rate:.2f})")

    Key Features:

    • Zero external dependencies: pure in-memory dict + dataclass, no Redis/ZooKeeper needed
    • Expertise matching algorithm: exact match 1.0 → fuzzy match 0.7 → no match 0.0
    • Session statistics: automatically tracks each agent's session count and success rate, providing weighted basis for distributed evolution
    • Default fallback: supports setting a general-purpose agent as fallback

    2. ExpertRouter¶

    "The right person for the right question"

    A three-tier routing strategy ensuring each question reaches the most suitable expert agent. No LLM involvement in routing decisions — zero API calls, zero latency.

    User Question
    │
    ▼
┌──────────────┐
│ L1: Keyword   │ ← 8 domains × 80+ keywords, millisecond-level
│   Match       │   Legal / Tech / Finance / Management / Product
│   Hit → Route │   Science / Education / Healthcare
└──────┬───────┘
       │ No match
       ▼
┌──────────────┐
│ L2: Semantic  │ ← Requires embedder, computes cosine similarity
│   Match       │   between question and expertise descriptions
│   sim≥0.3 → Route │
└──────┬───────┘
       │ No match
       ▼
┌──────────────┐
│ L3: Default   │ ← General-purpose agent as fallback
│   Fallback    │
└──────────────┘

router = ExpertRouter(registry, embedder=embedder)

agent, confidence, strategy = router.route(

    3. DistributedEvolver¶

    Each Agent evolves independently, periodically merged into global wisdom

    This is the most ambitious capability of v0.9.0. In traditional single-agent scenarios, the weights of 7 thinking laws could only be adjusted through a single agent's reflection. In v0.9.0, each expert agent maintains its own MetaEvolver instance (independent evolver_<agent_id>.db), evolving weights from their respective domain experience, then aggregating into global weights through merge strategies.

      Agent A (Legal)       Agent B (Tech)        Agent C (Management)
  Independent           Independent           Independent
  Evolution             Evolution             Evolution
  evolver_A.db          evolver_B.db          evolver_C.db
      │                     │                     │
      └─────────────────────┼─────────────────────┘
                            │
                    ┌───────▼───────┐
                    │  Global Merge │
                    │  3 Strategies │
                    └───────┬───────┘
                            │
                    ┌───────▼───────┐
                    │  Conflict     │
                    │  Arbitration  │
                    │  Divergence    │
                    │  >0.15: favor │
                    │  higher SR    │
                    └───────────────┘

    Three Merge Strategies:

    Strategy Formula Use Case
    simple_average Equal-weight average Homogeneous agent teams
    weighted_by_sessions Weighted by session count More experienced agents have more say
    weighted_by_success Weighted by success rate Recommended default — more effective agents have more say

    Conflict Arbitration Mechanism:

    When two agents adjust the same law's weight in opposite directions (one up, one down) and the divergence exceeds 0.15, arbitration is automatically triggered — favoring the agent with higher success rate, pulling toward the best weight by 60%.

    from soma.multi_agent import DistributedEvolver
from pathlib import Path

evo = DistributedEvolver(
    persist_dir=Path("./multi_agent_data"),
    default_weights={"first_principles": 0.9, "systems_thinking": 1.0, ...},
    merge_strategy="weighted_by_success",
)

# Register each agent's evolver
evo.register_agent("law_agent", law_agent.evolver, session_count=50, success_rate=0.85)
evo.register_agent("tech_agent", tech_agent.evolver, session_count=120, success_rate=0.92)
evo.register_agent("mgmt_agent", mgmt_agent.evolver, session_count=30, success_rate=0.78)

# Trigger all agents to evolve independently
changes = evo.evolve_all()
# → {"law_agent": [{"type": "weight_adjustment", ...}], ...}

# Merge into global weights
global_weights = evo.merge_weights()
# → {"first_principles": 0.91, "systems_thinking": 0.97, ...}

# Apply global weights to all agents
applied = evo.apply_all()
# → 3

# Statistics
print(evo.get_stats())
# → {"agent_count": 3, "merge_count": 1, "merge_strategy": "weighted_by_success", ...}

    4. ConsensusProtocol¶

    Enabling multiple Agents to reach consensus on the same question

    When a complex problem requires multiple experts to collectively assess from different angles, the consensus protocol provides three-tier strategies to coordinate disagreements and produce a unified conclusion.

    ┌─────────────────────────────────────────────────────┐
│              ConsensusProtocol                       │
│                                                      │
│  Input: Question + multiple AgentOpinions             │
│                                                      │
│  ┌──────────────┐                                    │
│  │ Auto-detect   │                                    │
│  │ agreement     │                                    │
│  │ level         │                                    │
│  │ ≥0.8 → Strong │                                    │
│  │ 0.5-0.8 →     │                                    │
│  │ Moderate       │                                    │
│  │ <0.5 → Severe  │                                    │
│  └──────┬───────┘                                    │
│         │                                            │
│  ┌──────▼──────────────────────────────────────┐     │
│  │ L1: Weighted Voting                          │     │
│  │   confidence × answer aggregation            │     │
│  │   For: simple questions, many opinions        │     │
│  ├──────────────────────────────────────────────┤     │
│  │ L2: LLM Arbitration                          │     │
│  │   LLM reads all opinions, resolves disputes  │     │
│  │   For: moderate disagreement, external view  │     │
│  ├──────────────────────────────────────────────┤     │
│  │ L3: Dialectical Synthesis                    │     │
│  │   Thesis → Antithesis → Synthesis            │     │
│  │   For: deep philosophical questions,         │     │
│  │   irreconcilable contradictions              │     │
│  └──────────────────────────────────────────────┘     │
│                                                      │
│  Output: ConsensusResult                              │
│    · consensus_answer: final unified answer           │
│    · agreement_level: consensus degree 0-1            │
│    · minority_view: minority opinion (if any)         │
│    · disagreements: detailed list of disagreement     │
│      points                                          │
└─────────────────────────────────────────────────────┘

    from soma.multi_agent import ConsensusProtocol, AgentOpinion

protocol = ConsensusProtocol(llm_call=my_llm_function)

# Each agent produces their opinion
opinions = [
    AgentOpinion(
        agent_id="law_agent",
        answer="From a legal perspective, the main risk in this contract is the IP ownership clause...",
        confidence=0.85,
        key_points=["Unclear IP ownership", "Asymmetric liability", "Disputed jurisdiction"],
    ),
    AgentOpinion(
        agent_id="tech_agent",
        answer="Technically, we need to focus on delivery standards and support terms...",
        confidence=0.78,
        key_points=["Vague acceptance criteria", "Undefined support SLA"],
    ),
    AgentOpinion(
        agent_id="mgmt_agent",
        answer="From a management perspective, the key risks are delivery timeline and change management...",
        confidence=0.72,
        key_points=["Unreasonable delivery timeline", "Missing change management process"],
    ),
]

# Form consensus
result = protocol.form_consensus("Review the comprehensive risks of this contract", opinions)

print(f"Agreement level: {result.agreement_level:.2f}")
print(f"Strategy: {result.strategy_used}")
# → strategy_used: "voting" (low divergence), "llm_arbitration" (moderate), "dialectical_synthesis" (severe)

if result.is_strong_consensus:
    print("Strong consensus reached!")
else:
    print(f"{len(result.disagreements)} disagreement points found")
    if result.minority_view:
        print(f"Minority view: {result.minority_view}")

    Agreement Level Detection Algorithm:

    1. Extract core claims (key_points) from all agent answers
    2. Compute Jaccard similarity of claim sets
    3. Weight by confidence (higher confidence agents have more weight)
    4. Output agreement_level ∈ [0, 1]

    5. Memory Isolation¶

    Three-State Isolation Model: Private / Group-Shared / Global

    v0.9.0 adds agent_id and shared_group_id fields to MemoryUnit, enabling finer-grained memory access control.

    ┌────────────────────────────────────────────┐
│              Memory Isolation               │
│                                             │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  │
│  │ Agent A  │  │ Agent B  │  │ Agent C  │  │
│  │ Private  │  │ Private  │  │ Private  │  │
│  │ agent=A  │  │ agent=B  │  │ agent=C  │  │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘  │
│       │              │              │       │
│       └──────────────┼──────────────┘       │
│                      │                      │
│              ┌───────▼───────┐              │
│              │ Group-Shared  │              │
│              │ group="team"  │              │
│              │ A,B,C visible │              │
│              └───────────────┘              │
│                                             │
│           ┌──────────────────┐              │
│           │   Global Memory   │              │
│           │   agent=""        │              │
│           │   all agents      │              │
│           └──────────────────┘              │
└────────────────────────────────────────────┘

    Isolation Enforcement in Query Paths:

    • FTS5 Keyword Search Path: SQL WHERE clause precisely filters agent_id = ? OR shared_group_id = ?
    • Vector Semantic Search Path: Python in-memory filtering mem.agent_id != agent_id → skip
    • Semantic Memory: namespace-encoded isolation (agent_id or user_id)
    • Skill Memory: user_id + agent/group dual-channel filtering

    Vector Index Startup Health Check (new in v0.9.0):

    When vector search is enabled after the fact (historical memories lack vectors), startup automatically detects indexed/total < 50% and triggers a full rebuild. This solves the upgrade scenario where "memories exist, but vectors don't" — causing semantic search to degrade to keyword search.

    Technical Details¶

    Zero External Dependencies¶

    All 4 new v0.9.0 modules are built entirely on SOMA's existing tech stack:

    Component Storage Computation Communication
    AgentRegistry Python dict O(1) lookup In-process
    ExpertRouter Python dict O(n) keyword scan In-process
    DistributedEvolver SQLite (per agent) Weighted average In-process
    ConsensusProtocol No persistence Jaccard + LLM In-process

    No Redis (registry center), RabbitMQ (message queue), gRPC (service communication), or etcd (configuration center) needed. All agents collaborate within a single Python process, suitable for:

    • Edge deployment (Raspberry Pi / embedded devices)
    • Single-machine multi-agent scenarios (customer service systems, education platforms)
    • Air-gapped environments (classified scenarios, offline terminals)

    Significance for Different Roles¶

    For AI Agent Developers¶

    From "Building Tools" to "Building Teams"

    Before v0.9.0, you used SOMA to equip a single agent with a thinking engine. After v0.9.0, you can create a team of expert agents — legal experts, technical advisors, management coaches — each with their own role, automatically assigned tasks through routing, and resolving disagreements through consensus protocols.

    Real-World Scenarios:

    • Enterprise AI Middle Platform: One SOMA instance managing 40 domain-specific avatars, each with independent memory and evolution paths, sharing a global knowledge base
    • Intelligent Customer Service: Frontend router automatically dispatches by question type to technical/after-sales/complaint experts, no LLM needed for intent recognition
    • Education Tutoring Platform: Math expert, language expert, science expert evolve independently, periodically merge teaching strategy weights

    What Doesn't Need to Change:

    • Existing soma.remember() / soma.respond() / soma.query_memory() APIs are completely unchanged
    • Single-agent usage is 100% compatible with v0.8.0
    • Only use soma.multi_agent module when you need multi-agent collaboration

    For Large Language Models (LLMs)¶

    From "One Model Answers Everything" to "Expert Models for Each Domain"

    v0.9.0's multi-agent architecture enables different LLM instances (or different prompt configurations of the same LLM) to play different roles:

    • Reduce Hallucinations: The consensus protocol's LLM arbitration layer cross-validates multiple independent judgments, reducing single-model bias
    • Save Tokens: The routing layer only sends questions to the most relevant Agent, rather than having all Agents participate
    • Break Through Single-Model Limitations: Different Agents can connect to different LLMs (Legal Agent using DeepSeek, Tech Agent using Claude, Management Agent using GPT), leveraging each model's strengths

    Key Insight: v0.9.0 doesn't make LLMs stronger — it makes the organization of LLMs smarter.

    For End Users¶

    From "Asking AI" to "Asking an Expert Team"

    When an average user asks "I want to start a business — what legal risks should I watch out for?":

    • Traditional AI: Gives generic advice, may miss critical legal details
    • v0.8.0 SOMA: Decomposes the question through 7 thinking laws, provides systematic analysis combining memory
    • v0.9.0 SOMA: Router automatically identifies "startup" → management, "legal risks" → legal, simultaneously engages both management expert and legal expert — both Agents analyze independently, then synthesize final advice through the consensus protocol

    Users don't need to know how many Agents are working behind the scenes — they just see a more comprehensive, more reliable answer that has been verified by multiple experts.

    From "Choosing a Library" to "Choosing an Architecture"

    5 Minutes: Create Your First Expert Team¶

    from soma import SOMA, SOMA_Agent, SOMAConfig
from soma.multi_agent import (
    AgentRegistry, ExpertRouter,
    DistributedEvolver, ConsensusProtocol, AgentOpinion,
)
from pathlib import Path

# 1. Create 3 expert Agents
config = SOMAConfig(
    framework_path="wisdom_laws.yaml",
    episodic_persist_dir=Path("./team_data"),
    use_vector_search=True,
)

law_agent = SOMA_Agent(config, agent_id="law_expert", group_id="team")
tech_agent = SOMA_Agent(config, agent_id="tech_architect", group_id="team")
mgmt_agent = SOMA_Agent(config, agent_id="mgmt_coach", group_id="team")

# 2. Register in registry
registry = AgentRegistry()
registry.register(law_agent, expertise=["legal", "contract", "compliance", "IP"],
                  description="Legal risk analysis expert")
registry.register(tech_agent, expertise=["tech", "architecture", "performance", "security"],
                  description="Technical architecture advisor")
registry.register(mgmt_agent, expertise=["management", "strategy", "team", "organization"],
                  description="Management coach", is_default=True)

# 3. Create router
router = ExpertRouter(registry, embedder=law_agent.embedder)

# 4. Route question to expert
expert, confidence, strategy = router.route(
    "How to review IP clauses in a SaaS contract?"
)
print(f"Routed to: {expert.agent_id} (confidence={confidence:.2f}, strategy={strategy})")
# → Routed to: law_expert (confidence=0.67, strategy=l1)

# 5. Answer with the routed expert
answer = expert.respond(
    "How to review IP clauses in a SaaS contract?",
    user_id="user_001"
)
print(answer)

    10 Minutes: Multi-Agent Consensus Decision¶

    # Multiple experts give opinions on the same complex question
question = "Should the company build an in-house AI team or outsource?"

opinions = []
for agent_id in ["law_expert", "tech_architect", "mgmt_coach"]:
    agent = registry.get(agent_id)
    response = agent.respond(question, user_id="ceo_001")
    # Extract core claims from response (simplified example)
    foci = agent.decompose(question)
    key_points = [f.dimension[:60] for f in foci[:3]]

    opinions.append(AgentOpinion(
        agent_id=agent_id,
        answer=response,
        confidence=0.8,
        foci=foci,
        key_points=key_points,
    ))

# Form consensus
protocol = ConsensusProtocol()
result = protocol.form_consensus(question, opinions)

print(f"Agreement level: {result.agreement_level:.2f}")
print(f"Strategy: {result.strategy_used}")
print(f"Final opinion: {result.consensus_answer[:200]}")
if result.minority_view:
    print(f"Minority view: {result.minority_view}")

    Complete API Reference¶

    AgentRegistry¶

    Method Description
    register(agent, expertise, description, is_default) Register an expert agent
    unregister(agent_id) Unregister an agent
    get(agent_id) Get agent instance
    get_default() Get default fallback agent
    find_experts(domain, min_score) Find matching experts by domain
    list_agents() List all agent metadata
    record_session(agent_id, success) Record session result, update stats
    agent_count Number of registered agents

    ExpertRouter¶

    Method Description
    route(problem, min_confidence, allow_fallback) Single-expert routing, returns (agent, confidence, strategy)
    route_multi(problem, top_k, min_confidence) Multi-expert routing, returns top-K matches
    stats Routing statistics: route_count, l1_hit_rate, l2_hit_rate, fallback_rate

    DistributedEvolver¶

    Production Recommendations¶

    Production-Verified Configuration (零熵智库)¶

    The following configuration has been validated in real production (40 avatars, 1,752 memories, 34 days continuous operation):

    soma = SOMA(
    llm="openai/deepseek-chat",
    use_vector_search=True,
    recall_threshold=0.01,   # Low threshold, sorted by RRF fusion
    top_k=15,                 # Moderate candidate pool size
    persist_dir="/opt/soma_data",
)

    Performance Guidelines¶

    • < 100 memories: No special optimization needed
    • 100 - 2,000 memories: Recommended config (current production experience)
    • > 2,000 memories: Consider raising recall_threshold, lowering top_k, increasing evolve() interval
    • Multi-Agent scenarios: Separate directories per agent to avoid SQLite lock contention

    Upgrading from v0.8.0¶

    pip install --upgrade soma-wisdom>=0.9.0

    Zero code changes — all v0.8.0 APIs are backward compatible. Multi-agent capabilities are provided through the new soma.multi_agent module and do not affect existing code.

    On startup, if historical memories are detected to lack vector indices (indexed/total < 50%), a full rebuild is automatically triggered. The first startup may take 2-5 minutes (depending on memory count); subsequent startups complete instantly.
