    版本概览¶

    SOMA v0.9.0 是 Somatic Wisdom Architecture 自 v0.1.0 以来最大的一次架构升级。此前 v0.8.0 完成了个体智能体的认知推理闭环（因果推理、冲突检测、跨域类比、质量评估），v0.9.0 则实现了从个体到群体的跃迁——让多个 SOMA Agent 以团队形式协作，各自拥有独立记忆、独立进化路径和专长领域，通过专家路由、分布式演化和共识协议形成超越个体的集体智能。

    版本演进路线¶

    v0.5.0 — 基础框架：7条思维规律 + 向量检索 + 进化闭环
v0.6.0 — 推理模板：结构化推理框架 + 基准测试体系
v0.7.0 — 记忆智能：合并摘要 + 主动遗忘 + 外部知识导入
v0.8.0 — 认知推理：因果链 + 冲突检测 + 跨域类比 + 质量评估
v0.9.0 — 群体智能：多Agent协作 + 专家路由 + 分布式演化 + 共识协议 ← 当前

    五大核心能力¶

    一、Agent 注册表（AgentRegistry）¶

    领域专长的形式化表达

    每个 Agent 不再是同质的「通用思考者」，而是拥有明确的专长领域标签、成功率追踪和会话统计的专业智能体。

    from soma.multi_agent import AgentRegistry, AgentInfo

registry = AgentRegistry()

# 注册法律专家 agent
registry.register(
    agent=law_agent,
    expertise=["法律", "合同", "诉讼", "知识产权"],
    description="擅长法律条文分析、合同审核与合规判断",
    is_default=False,
)

# 注册技术架构师 agent
registry.register(
    agent=tech_agent,
    expertise=["技术", "架构", "性能优化", "系统设计"],
    description="擅长分布式系统架构、性能瓶颈分析与技术选型",
)

# 按领域查找专家
experts = registry.find_experts("法律")
# → [(law_agent, 1.0), ...]

# 获取所有agent统计
for info in registry.list_agents():
    print(f"{info.agent_id}: {info.expertise} "
          f"(sessions={info.session_count}, sr={info.success_rate:.2f})")

    关键特性：

    • 零外部依赖：纯内存字典 + dataclass，无需 Redis/ZooKeeper
    • 专长匹配算法：精确匹配 1.0 → 模糊匹配 0.7 → 不匹配 0.0
    • 会话统计：自动追踪每个 agent 的处理会话数和成功率，为分布式演化提供加权依据
    • 默认回退：支持设置一个通用 agent 作为兜底

    二、专家路由器（ExpertRouter）¶

    "找对的人回答对的问题"

    三层路由策略，确保每个问题交给最合适的专家 agent 处理。不需要 LLM 参与路由决策——零 API 调用、零延迟。

    用户问题
    │
    ▼
┌──────────────┐
│ L1: 关键词匹配 │ ← 8大领域 × 80+关键词，毫秒级
│ 命中 → 直接路由 │   法律/技术/金融/管理/产品/科学/教育/医疗
└──────┬───────┘
       │ 未命中
       ▼
┌──────────────┐
│ L2: 语义匹配   │ ← 需要 embedder，计算问题 vs 专长描述的余弦相似度
│ 相似度≥0.3 → 路由 │
└──────┬───────┘
       │ 未命中
       ▼
┌──────────────┐
│ L3: 默认回退   │ ← 通用 agent 兜底
└──────────────┘

    L1 领域关键词表：

    领域 关键词示例
    法律 法律、合同、诉讼、法规、合规、知识产权、版权、专利、仲裁
    技术 代码、架构、API、数据库、部署、算法、性能、bug、编程
    金融 投资、股票、基金、财务、预算、成本、收入、利润、税务
    管理 团队、管理、领导、组织、战略、流程、考核、招聘、文化
    产品 用户、需求、体验、设计、功能、迭代、增长、留存、转化
    科学 实验、假设、数据、统计、变量、控制组、显著性、相关性
    教育 学习、教学、课程、培训、知识、技能、考试、毕业
    医疗 诊断、治疗、药物、手术、患者、临床、病理、预防 
    from soma.multi_agent import ExpertRouter

router = ExpertRouter(registry, embedder=embedder)

# 单专家路由
agent, confidence, strategy = router.route("如何审查一份技术合同中的知识产权条款？")
# → (law_agent, 0.67, "l1")  # 命中"法律"+"合同"+"知识产权"等关键词

# 多专家路由（用于多角度分析）
agents = router.route_multi("公司数字化转型的战略规划", top_k=3)
# → [(tech_agent, 0.67, "l1"), (mgmt_agent, 0.67, "l1"), (product_agent, 0.33, "l1")]

# 路由统计
print(router.stats)
# → {"route_count": 100, "l1_hit_rate": 0.73, "l2_hit_rate": 0.15, "fallback_rate": 0.12}

    三、分布式权重演化（DistributedEvolver）¶

    每个 Agent 独立演化，定期合并为全局智慧

    这是 v0.9.0 最具野心的能力。在传统单一 agent 场景中，7 条思维规律的权重只能通过一个 agent 的反思来调整。在 v0.9.0 中，每个专家 agent 维护自己的 MetaEvolver 实例（独立的 evolver_<agent_id>.db），从各自的领域经验中演化权重，然后通过合并策略聚合成全局权重。

      Agent A (法律)        Agent B (技术)        Agent C (管理)
  独立演化              独立演化              独立演化
  evolver_A.db          evolver_B.db          evolver_C.db
      │                     │                     │
      └─────────────────────┼─────────────────────┘
                            │
                    ┌───────▼───────┐
                    │  全局权重合并  │
                    │  3种合并策略   │
                    └───────┬───────┘
                            │
                    ┌───────▼───────┐
                    │  冲突仲裁      │
                    │  分歧>0.15时   │
                    │  偏向成功率高的 │
                    └───────────────┘

    三种合并策略：

    策略 公式 适用场景
    simple_average 等权平均 同质 agent 团队
    weighted_by_sessions 按会话数加权 经验丰富的 agent 更有话语权
    weighted_by_success 按成功率加权 推荐默认——效果好的 agent 更有话语权

    冲突仲裁机制：

    当两个 agent 对同一条规律的权重调整方向相反（一个上调、一个下调），且分歧超过 0.15 时，自动触发仲裁——以成功率高的 agent 为准，向最佳权重靠拢 60%。

    from soma.multi_agent import DistributedEvolver

evo = DistributedEvolver(
    persist_dir=Path("./multi_agent_data"),
    default_weights={"first_principles": 0.9, "systems_thinking": 1.0, ...},
    merge_strategy="weighted_by_success",
)

# 注册各agent的演化器
evo.register_agent("law_agent", law_agent.evolver, session_count=50, success_rate=0.85)
evo.register_agent("tech_agent", tech_agent.evolver, session_count=120, success_rate=0.92)
evo.register_agent("mgmt_agent", mgmt_agent.evolver, session_count=30, success_rate=0.78)

# 触发所有agent独立演化
changes = evo.evolve_all()
# → {"law_agent": [{"type": "weight_adjustment", ...}], ...}

# 合并为全局权重
global_weights = evo.merge_weights()
# → {"first_principles": 0.91, "systems_thinking": 0.97, ...}

# 将全局权重应用到所有agent
applied = evo.apply_all()
# → 3

# 统计信息
print(evo.get_stats())
# → {"agent_count": 3, "merge_count": 1, "merge_strategy": "weighted_by_success", ...}

    四、共识形成协议（ConsensusProtocol）¶

    让多个 Agent 就同一问题达成共识

    当一个复杂问题需要多个专家从不同角度共同判断时，共识协议提供了三级策略来协调分歧产出统一结论。

    ┌─────────────────────────────────────────────────────┐
│              ConsensusProtocol                       │
│                                                      │
│  输入: 问题 + 多个AgentOpinion                        │
│                                                      │
│  ┌──────────────┐                                    │
│  │ agreement_level 自动检测 │                         │
│  │ ≥0.8 → 强共识（直接用）  │                         │
│  │ 0.5-0.8 → 中等分歧       │                         │
│  │ <0.5 → 严重分歧          │                         │
│  └──────┬───────┘                                    │
│         │                                            │
│  ┌──────▼──────────────────────────────────────┐     │
│  │ L1: weighted_voting                          │     │
│  │   加权投票：confidence×answer合并             │     │
│  │   适用：简单问题、观点数量多                   │     │
│  ├──────────────────────────────────────────────┤     │
│  │ L2: llm_arbitration                          │     │
│  │   LLM阅读所有观点后裁决分歧                    │     │
│  │   适用：中等分歧、需要外部判断                 │     │
│  ├──────────────────────────────────────────────┤     │
│  │ L3: dialectical_synthesis                    │     │
│  │   正→反→合 辩证综合                           │     │
│  │   适用：深层哲学问题、不可调和的矛盾           │     │
│  └──────────────────────────────────────────────┘     │
│                                                      │
│  输出: ConsensusResult                                │
│    · consensus_answer: 最终共识回答                    │
│    · agreement_level: 共识度 0-1                       │
│    · minority_view: 少数派观点（如有）                 │
│    · disagreements: 分歧点详细列表                     │
└─────────────────────────────────────────────────────┘

    from soma.multi_agent import ConsensusProtocol, AgentOpinion

protocol = ConsensusProtocol(llm_call=my_llm_function)

# 各agent产出观点
opinions = [
    AgentOpinion(
        agent_id="law_agent",
        answer="从法律角度看，此合同的风险主要在知识产权归属条款...",
        confidence=0.85,
        key_points=["知识产权归属不明确", "违约责任不对等", "管辖权条款有争议"],
    ),
    AgentOpinion(
        agent_id="tech_agent",
        answer="技术上需要关注交付标准和技术支持条款...",
        confidence=0.78,
        key_points=["交付物验收标准模糊", "技术支持时效未定义"],
    ),
    AgentOpinion(
        agent_id="mgmt_agent",
        answer="从管理角度，关键风险是交付周期和变更管理流程...",
        confidence=0.72,
        key_points=["交付周期不合理", "变更管理流程缺失"],
    ),
]

# 形成共识
result = protocol.form_consensus("审查此合同的综合风险", opinions)

print(f"共识度: {result.agreement_level:.2f}")
print(f"策略: {result.strategy_used}")
# → strategy_used: "voting" (分歧小时), "llm_arbitration" (中等), "dialectical_synthesis" (严重)

if result.is_strong_consensus:
    print("强共识达成!")
else:
    print(f"存在 {len(result.disagreements)} 处分歧")
    if result.minority_view:
        print(f"少数派: {result.minority_view}")

    共识度自检算法：

    1. 提取所有 agent 答案的核心论点（key_points）
    2. 计算论点集合的 Jaccard 相似度
    3. 按置信度加权（高置信度 agent 的论点权重更高）
    4. 综合输出 agreement_level ∈ [0, 1]

    五、记忆隔离（Memory Isolation）¶

    三态隔离模型：私有 / 组共享 / 全局

    v0.9.0 为 MemoryUnit 新增了 agent_id 和 shared_group_id 字段，实现了更精细的记忆访问控制。

    ┌────────────────────────────────────────────┐
│              Memory Isolation               │
│                                             │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  │
│  │ Agent A  │  │ Agent B  │  │ Agent C  │  │
│  │ 私有记忆 │  │ 私有记忆 │  │ 私有记忆 │  │
│  │ agent=A  │  │ agent=B  │  │ agent=C  │  │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘  │
│       │              │              │       │
│       └──────────────┼──────────────┘       │
│                      │                      │
│              ┌───────▼───────┐              │
│              │  组共享记忆    │              │
│              │  group="team" │              │
│              │  A,B,C 均可见  │              │
│              └───────────────┘              │
│                                             │
│           ┌──────────────────┐              │
│           │   全局记忆        │              │
│           │   agent=""        │              │
│           │   所有agent可见   │              │
│           └──────────────────┘              │
└────────────────────────────────────────────┘

    查询时的隔离执行路径：

    • FTS5 关键词搜索路径：SQL WHERE 子句精确过滤 agent_id = ? OR shared_group_id = ?
    • 向量语义搜索路径：Python 内存过滤 mem.agent_id != agent_id → skip
    • 语义记忆：namespace 编码隔离（agent_id or user_id）
    • 技能记忆：user_id + agent/group 双通道过滤

    向量索引起始健康检查（v0.9.0 新增）：

    当向量搜索后期启用时（历史记忆中无向量），启动时自动检测 indexed/total < 50% 并触发全量重建。解决了升级场景下「有记忆、无向量」导致语义搜索退化为关键词搜索的问题。

    技术细节¶

    零外部依赖¶

    v0.9.0 的 4 个新模块全部基于 SOMA 现有技术栈：

    组件 存储 计算 通信
    AgentRegistry Python dict O(1) 查找 进程内
    ExpertRouter Python dict O(n) 关键词扫描 进程内
    DistributedEvolver SQLite（每agent独立） 加权平均 进程内
    ConsensusProtocol 无持久化 Jaccard + LLM 进程内

    无需引入 Redis（注册中心）、RabbitMQ（消息队列）、gRPC（服务通信）、etcd（配置中心）。全部在单个 Python 进程中协作，适合：

    • 边缘部署（树莓派 / 嵌入式设备）
    • 单机多 Agent 场景（客服系统、教育平台）
    • 无网络环境（涉密场景、离线终端）

    对不同角色的意义¶

    对 AI 智能体开发者¶

    从「造工具」到「建团队」

    v0.9.0 之前，你用 SOMA 给单个 Agent 装上思考引擎。v0.9.0 之后，你可以创建一支 Agent 专家团队——法律专家、技术顾问、管理教练各司其职，通过路由自动分配任务，通过共识协议协调分歧。

    实际场景：

    • 企业 AI 中台：一个 SOMA 实例管理 40 个不同领域的分身，每个分身拥有独立记忆和演化路径，共享全局知识库
    • 智能客服系统：前端路由器根据问题类型自动分派给技术/售后/投诉专家，不需要 LLM 做意图识别
    • 教育辅导平台：数学专家、语文专家、科学专家独立演化，定期合并教学策略权重

    什么不需要改变：

    • 现有的 soma.remember() / soma.respond() / soma.query_memory() API 完全不变
    • 单个 Agent 的使用方式与 v0.8.0 完全兼容
    • 只有当你需要多 Agent 协作时才使用 soma.multi_agent 模块

    对大模型（LLM）¶

    从「一个模型回答一切」到「专家模型各司其职」

    v0.9.0 的多 Agent 架构让不同的 LLM 实例（或同一 LLM 的不同 prompt 配置）扮演不同角色：

    • 降低幻觉：共识协议的 LLM 仲裁层让多个独立判断交叉验证，减少单一模型的偏见
    • 节省 Token：路由层只把问题发给最相关的 Agent，而不是让所有 Agent 都参与
    • 突破单一模型的局限：不同 Agent 可以接入不同的 LLM（法律 Agent 用 DeepSeek、技术 Agent 用 Claude、管理 Agent 用 GPT），取长补短

    关键洞察：v0.9.0 不是让 LLM 更强，而是让 LLM 的组织方式更聪明。

    对终端用户¶

    从「问 AI」到「问专家团队」

    当一个普通用户提出「我想创业，应该注意什么法律风险？」时：

    • 传统 AI：给出通用建议，可能遗漏关键法律细节
    • v0.8.0 SOMA：通过 7 条思维规律拆解问题，结合记忆库给出系统分析
    • v0.9.0 SOMA：路由器自动识别「创业」→管理，「法律风险」→法律，同时调动管理专家和法律专家，两个 Agent 各自分析后通过共识协议合成最终建议

    用户不用关心背后有几个 Agent 在工作——他们只看到一个经过多专家验证、更全面可靠的回答。

    对技术决策者¶

    从「选一个库」到「选一个架构」

    SOMA 不是另一个向量数据库，不是另一个 Agent 框架，而是一个完整的认知架构：

    层级 SOMA 提供的 传统方案需要的
    推理 7 条思维规律 + 推理模板 手写 prompt 工程
    记忆 SQLite + 向量 + FTS5 + 三库 多组件拼接（ChromaDB + PostgreSQL + Redis）
    进化 自动权重调整 + 合并/遗忘 无（或手写规则）
    协作 注册表 + 路由器 + 共识协议 ZooKeeper + 消息队列 + 手写协议
    部署 pip install soma-wisdom Docker Compose / K8s 多服务编排

    单二进制启动，零运维负担。

    快速开始¶

    安装¶

    pip install soma-wisdom>=0.9.0

    5 分钟：创建你的第一个专家团队¶

    from soma import SOMA, SOMA_Agent, SOMAConfig
from soma.multi_agent import (
    AgentRegistry, ExpertRouter,
    DistributedEvolver, ConsensusProtocol, AgentOpinion,
)
from pathlib import Path

# 1. 创建 3 个专家 Agent
config = SOMAConfig(
    framework_path="wisdom_laws.yaml",
    episodic_persist_dir=Path("./team_data"),
    use_vector_search=True,
)

law_agent = SOMA_Agent(config, agent_id="law_expert", group_id="team")
tech_agent = SOMA_Agent(config, agent_id="tech_architect", group_id="team")
mgmt_agent = SOMA_Agent(config, agent_id="mgmt_coach", group_id="team")

# 2. 注册到注册表
registry = AgentRegistry()
registry.register(law_agent, expertise=["法律", "合同", "合规", "知识产权"],
                  description="法律风险分析专家")
registry.register(tech_agent, expertise=["技术", "架构", "性能", "安全"],
                  description="技术架构顾问")
registry.register(mgmt_agent, expertise=["管理", "战略", "团队", "组织"],
                  description="管理教练", is_default=True)

# 3. 创建路由器
router = ExpertRouter(registry, embedder=law_agent.embedder)

# 4. 路由问题到专家
expert, confidence, strategy = router.route(
    "如何审查一份SaaS合同中的知识产权条款？"
)
print(f"路由到: {expert.agent_id} (置信度={confidence:.2f}, 策略={strategy})")
# → 路由到: law_expert (置信度=0.67, 策略=l1)

# 5. 用路由到的专家回答问题
answer = expert.respond("如何审查一份SaaS合同中的知识产权条款？",
                        user_id="user_001")
print(answer)

    10 分钟：多 Agent 共识决策¶

    # 对同一个复杂问题，多个专家给出观点
question = "公司应该自建AI团队还是外包？"

opinions = []
for agent_id in ["law_expert", "tech_architect", "mgmt_coach"]:
    agent = registry.get(agent_id)
    response = agent.respond(question, user_id="ceo_001")
    # 从回答中提取核心论点（简化示例）
    foci = agent.decompose(question)
    key_points = [f.dimension[:60] for f in foci[:3]]

    opinions.append(AgentOpinion(
        agent_id=agent_id,
        answer=response,
        confidence=0.8,
        foci=foci,
        key_points=key_points,
    ))

# 形成共识
protocol = ConsensusProtocol()
result = protocol.form_consensus(question, opinions)

print(f"共识度: {result.agreement_level:.2f}")
print(f"策略: {result.strategy_used}")
print(f"最终意见: {result.consensus_answer[:200]}")
if result.minority_view:
    print(f"少数派意见: {result.minority_view}")

    完整 API 参考¶

    AgentRegistry¶

    方法 说明
    register(agent, expertise, description, is_default) 注册专家 agent
    unregister(agent_id) 注销 agent
    get(agent_id) 获取 agent 实例
    get_default() 获取默认回退 agent
    find_experts(domain, min_score) 按领域查找匹配的专家
    list_agents() 列出所有 agent 元数据
    record_session(agent_id, success) 记录会话结果，更新统计
    agent_count 已注册 agent 数量

    ExpertRouter¶

    方法 说明
    route(problem, min_confidence, allow_fallback) 单专家路由，返回 (agent, confidence, strategy)
    route_multi(problem, top_k, min_confidence) 多专家路由，返回前 K 个匹配
    stats 路由统计：route_count, l1_hit_rate, l2_hit_rate, fallback_rate

    DistributedEvolver¶

    方法 说明
    register_agent(agent_id, evolver, session_count, success_rate) 注册 agent 演化器
    unregister_agent(agent_id) 注销
    evolve_agent(agent_id) 触发单个 agent 演化
    evolve_all() 触发所有 agent 独立演化
    merge_weights(strategy) 执行全局权重合并（3种策略可选）
    get_global_weights() 获取最近一次合并的全局权重
    apply_global_weights(agent_id) 将全局权重应用到指定 agent
    apply_all() 将全局权重应用到所有 agent
    get_stats() 分布式演化统计

    ConsensusProtocol¶

    方法 说明
    form_consensus(question, opinions) 形成共识，返回 ConsensusResult
    ConsensusResult.is_strong_consensus 共识是否强（≥0.8）
    ConsensusResult.agent_count 参与 agent 数量
    ConsensusResult.minority_view 少数派观点（如有）

    MemoryUnit（新增字段）¶

    字段 类型 说明
    agent_id str 记忆所属 agent 的 ID
    shared_group_id str 记忆所属协作组的 ID
    user_id str 记忆所属用户的 ID（已有）

    生产环境建议¶

    零熵智库验证配置¶

    以下配置已在实际生产环境（40个分身、1,752条记忆、34天连续运行）中验证：

    soma = SOMA(
    llm="openai/deepseek-chat",
    use_vector_search=True,
    recall_threshold=0.01,   # 低阈值，由 RRF 融合排序
    top_k=15,                 # 候选池适中的数据量
    persist_dir="/opt/soma_data",
)

    性能建议¶

    • < 100 条记忆：无需特别优化
    • 100 - 2,000 条：推荐配置（当前生产经验）
    • > 2,000 条：考虑调高 recall_threshold、降低 top_k、增加 evolve() 间隔
    • 多 Agent 场景：每个 Agent 独立目录避免 SQLite 锁竞争

    从 v0.8.0 升级¶

    pip install --upgrade soma-wisdom>=0.9.0

    零代码改动——所有 v0.8.0 API 向后兼容。多 Agent 能力是通过新增的 soma.multi_agent 模块提供的，不会影响现有代码。

    启动时如果检测到历史记忆缺少向量索引（indexed/total < 50%），会自动触发全量重建。首次启动可能耗时 2-5 分钟（取决于记忆数量），后续启动瞬间完成。

    展望 v0.10.0¶

    v0.9.0 完成了个体智能到群体智能的跃迁。v0.10.0 的探索方向：

    • Agent 自动孵化：当路由器发现某个领域的回退率过高时，自动创建新的专家 Agent
    • 跨 Agent 知识蒸馏：高成功率 Agent 的经验自动迁移到低成功率 Agent
    • Agent 市场协议：Agent 之间的知识交易和信誉系统
    • 联邦学习式演化：多个 SOMA 实例（不同服务器）间安全共享演化权重，不共享原始记忆
