    SOMA v0.9.1 零熵智慧整合方案¶

    版本定位: v0.9.0 里程碑升级 | 原则: 零接口变更 | 新增: 觉察维度

    一、为什么是 v0.9.1¶

    版本语义¶

    v0.9.0 完成了「从个体到群体」的跃迁 — 多 Agent 协作、专家路由、共识协议已就位。本次整合在此之上新增一个维度：觉察。

    不改变现有架构。不修改任何公开 API。六个模块像六块积木，按需叠加上去。

    v0.9.1 = v0.9.0 的全部能力 + 「看见自己在怎么想」的觉察层。

    与 v0.10.0 的关系¶

    原计划 v0.10.0 的四个方向（Agent 自动孵化、跨 Agent 知识蒸馏、Agent 市场协议、联邦演化）不受影响。本次整合是中途插入的里程碑，不替代原有路线，可并行推进。

    二、兼容性第一原则¶

    2.1 硬约束¶

    2000+ 开发者的存量代码，升级到 v0.9.1 后不做任何改动就能正常运行。

    这是最高优先级约束。具体体现为五条铁律：

    # 铁律 验证方式
    1 所有公开方法的签名不变 逐个比对 SOMA、SOMA_Agent、SOMAConfig、ConsensusProtocol 等 30+ 个公开方法
    2 所有公开类的构造参数不变 现有 SOMA(llm="...", use_vector_search=True) 写法不报错
    3 所有公开方法的返回类型不变 respond() 还是返回 str，chat() 还是返回 dict
    4 所有新行为默认关闭 新增 config 字段默认值 False / None / 0
    5 新模块不影响已有 import 新增 soma.entropy、soma.hooks 等包，不修改已有 import 路径

    三、全局约束与设计准则¶

    每个模块的设计必须满足以下 8 条准则：

    G1. 错误隔离：任何新模块的异常不得传播到核心管道。所有新代码调用包裹在 try/except 中，记录日志后静默跳过。

    G2. 无嵌入器降级：依赖 embedder 的功能必须有非嵌入器的回退路径（如字符级 bigram 相似度替代余弦相似度）。

    G3. 无 LLM 降级：依赖 LLM 的功能必须有非 LLM 的回退路径（如规则替代仲裁）。

    G4. 可配置开关：每个模块有独立的启用开关。用户可以只开某一个而关闭其他五个。

    G5. 资源上限：任何新数据存储（SQLite 表、内存缓存）都有明确的大小上限和淘汰策略。

    G6. 线程安全：在 WAL 模式下读写不互斥。Python 层的全局状态（如 hook registry）使用线程安全的数据结构。

    G7. 日志可观测：每个模块的关键状态变化（启用/禁用/触发/异常）都输出 INFO/DEBUG 日志。

    G8. 独立可测试：每个模块可以在 SOMA(enable_xxx=False) 环境下独立测试，不需要同时启用其他模块。

    四、模块逐一拆解¶

    以下每个模块按统一格式展开： - 实现内容（精确到新增哪些文件、修改哪些文件的哪些位置） - 兼容性保证（证明对存量零影响） - 风险矩阵（每个步骤可能出什么错、概率、影响、缓解）

    风险矩阵¶

    # 故障场景 触发条件 概率 影响 缓解措施
    E1 negative_words.txt 文件缺失 安装不完整 低 熵值计算失败 except FileNotFoundError → 使用内置 50 词最小词表
    E2 旧版 analytics 表无 entropy_score 列 从 v0.9.0 升级 中 写入失败 ALTER TABLE 用 try/except 包裹，列已存在则跳过
    E3 对话轮次不足（<3 轮） 新用户 高 无法计算连贯性 len(turns) < 3 → 返回 clarity_index=50, grade="数据不足"
    E4 embedder 不可用 未安装 ONNX 低 连贯性计算崩溃 回退到字符 bigram 重叠率（已有实现可复用）
    E5 10000+ 轮对话后窗口计算变慢 长期用户 低 响应延迟 窗口固定 20 轮，O(20) 计算量恒定

    兼容性证明¶

    检查项 结果
    现有方法签名是否改变 否

    关键设计决策 — 钩子输出的三通道（D1 确认：C+A 混合方案）：

    钩子提示绝不混入 respond() 的返回值。通过三条独立通道输出：

    Channel 1: soma.reflections (内存 list) — 外部应用主动拉取
  → soma.get_hook_reflections(user_id, limit) → List[dict]
  → 每个 reflection: {step, hook_name, prompt, timestamp, session_id}
  → 外部应用决定是否展示、如何展示给用户

Channel 2: Python logging (INFO 级别) — 静默日志
  → 格式: [HOOK:{hook_name}] step={step} {prompt}
  → 写入日志文件，不打扰用户

Channel 3: reflection_log 表 (SQLite 持久化)
  → reflection_type="inner_hook"
  → 可被 MetaEvolver.evolve() 读取用于权重调整

    阴阳权重自动演化（D3 确认参数）¶

USER_TYPE_ADJUSTMENTS = {
    "entrepreneur": {"action": +0.1, "awareness": -0.1},
    "practitioner": {"action": -0.1, "awareness": +0.1},
    "general":       {"action": 0.0, "awareness": 0.0},
}

# 每次 evolve() 调用的自动调整（步进 ±0.02，硬边界 0.3/0.7）
def _auto_adjust_zhongdao(self, recent_feedback: list) -> None:
    avg_feedback = sum(r.score for r in recent_feedback) / len(recent_feedback)
    if avg_feedback > 0.7:
        # 正向反馈 → 强化当前主导方向，但步进极小
        delta = min(0.02, avg_feedback - 0.7)
    elif avg_feedback < 0.3:
        # 负向反馈 → 回调
        delta = -min(0.02, 0.3 - avg_feedback)
    else:
        return  # 中性区间 [0.3, 0.7]，不调整

    self.action_weight = clamp(self.action_weight + delta, 0.3, 0.7)
    self.awareness_weight = 1.0 - self.action_weight  # 保持总和恒为 1.0

    第二关：5 柱接受量表（人工审核，每柱 0-20 分，总分 ≥ 70 分通过）

    柱名 含义 满分标准
    善行 是否引向正向行动 提供具体、可操作的向善行动路径
    善人 是否促进人的成长 关注人的内在成长而非仅外部结果
    善心 是否引导内求觉察 包含自我审视和反思的触发点
    善史 是否符合长期价值 不追求短期效果，考虑长期影响
    善修 是否可实践验证 规律可以被个人实践检验，非纯理论

    八、总结¶

    这次整合的本质¶

    不是给 SOMA 添加六个「功能」。而是为其注入一个新的认知操作系统。

    当前 SOMA 的操作系统是「问题解决模式」——接收问题→拆解→检索→推理→回答。这次整合后，SOMA 将获得第二个模式——「觉察辅助模式」——在解决问题的同时，温和地帮助使用者看见自己的思维惯性、情绪扰动、认知边界。

    决策状态¶

    6 项关键决策（D1-D6）已经与文化赋能师确认，详见第九节。核心决策： - D1：钩子走独立反射 API，不混入 answer 文本 - D2：连续 3 轮 < 40 分触发引导模式，2 轮 ≥ 50 分退出 - D3：阴阳权重 0.5/0.5 起，用户类型 ±0.1，自动演化 ±0.02，边界 0.3/0.7 - D4：4 类禁止 + 5 柱量表，需编写书面审核标准 - D5：高能量规律（≥50 或 >30 次采纳）需 3 人审核，旧版归档不删除 - D6：SOMA 熵值与零熵测试系统独立并行，v0.9.1 不做技术集成

    技术上的承诺¶

    承诺 如何保证
    不破坏一行存量代码 所有新功能默认关闭，新行为由独立 config flag 控制
    不改变任何一个 API 30 个公开接口签名和返回类型不变
    出问题可以一键回退 pip install 旧版本，数据库无损
    六个模块独立可开关 每个模块有独立的 enable_xxx config flag

    九、六项关键决策（已确认）¶

    以下 6 项决策经与文化赋能师深入讨论后已确定，作为模块实现的最终技术参数。

    D1：内求复盘钩子的输出通道¶

    决策：C+A 混合方案 — 独立钩子反射 API + 静默日志

    钩子提示绝不混入 respond() 的返回值。技术实现：

    钩子触发后的三条输出通道：
┌─────────────────────────────────────────────────────┐
│ Channel 1: soma.reflections (内存 list)              │
│   → soma.get_hook_reflections(user_id) → List[dict] │
│   → 外部应用主动拉取，按需展示给用户                   │
│   → 每个 reflection 包含: step, hook_name, prompt,   │
│      timestamp, session_id                          │
├─────────────────────────────────────────────────────┤
│ Channel 2: Python logging (级别 INFO)                │
│   → 静默写入日志文件，不打扰用户                       │
│   → 格式: [HOOK:{hook_name}] step={step} {prompt}   │
├─────────────────────────────────────────────────────┤
│ Channel 3: reflection_log 表 (持久化)                 │
│   → reflection_type="inner_hook"                    │
│   → 可被 MetaEvolver 的 evolve() 读取用于权重调整      │
└─────────────────────────────────────────────────────┘

    新增公开方法（不影响存量）：

    def get_hook_reflections(self, user_id: str = "", limit: int = 10) -> list:
    """获取最近的内求钩子提示，按时间倒序"""
    ...

def clear_hook_reflections(self, user_id: str = "") -> None:
    """清空已读取的钩子提示"""
    ...

    设计理由：钩子提示是「觉察辅助」而非「答案修正」。强行混入 answer 会破坏返回格式兼容性，让外部应用自行决定如何展示是最灵活的做法。

    D2：认知熵值的干预阈值¶

    决策：连续 3 轮 < 40 分触发引导模式，连续 2 轮 ≥ 50 分后退出

    引导模式的行为变更：

    正常模式 引导模式
    直接回答占 70%+ 直接回答降至 < 30%
    主动给出结论 以引导性问题为主（「你觉得…？」「如果换个角度…？」）
    无干预 响应末尾追加呼吸提醒（随机从 3 条模板中选取）

    技术参数：

    GUIDANCE_ENTER_THRESHOLD = 40      # 熵值低于 40 分
GUIDANCE_ENTER_CONSECUTIVE = 3     # 连续 3 轮
GUIDANCE_EXIT_THRESHOLD = 50       # 熵值恢复到 50 分以上
GUIDANCE_EXIT_CONSECUTIVE = 2      # 连续 2 轮
GUIDANCE_DIRECT_ANSWER_RATIO = 0.3 # 引导模式下直接回答占比上限

# 呼吸提醒模板（内置 3 条，可扩展）
BREATHING_REMINDERS = [
    "在继续之前，不妨先做一次深呼吸，感受当下的状态。",
    "暂停三秒，问问自己：我现在的情绪在哪个位置？",
    "注意到思绪的速度了吗？先回到呼吸，再回到问题。",
]

    引导模式状态存储在 CognitiveEntropyCalculator 的内存状态中，不持久化（会话级别的临时状态）。

    D3：守中执一算法的阴阳权重¶

    决策：初始 0.5/0.5，按用户类型微调，步进 ±0.02，硬边界 0.3/0.7

    # 初始权重
zhongdao_action_weight: float = 0.5     # 行动导向（阳）
zhongdao_awareness_weight: float = 0.5  # 觉察导向（阴）

# 用户类型微调（初始化时一次性调整）
USER_TYPE_ADJUSTMENTS = {
    "entrepreneur": {"action": +0.1, "awareness": -0.1},  # 创业者 → 偏行动
    "practitioner": {"action": -0.1, "awareness": +0.1},  # 修行者 → 偏觉察
    "general":       {"action": 0.0, "awareness": 0.0},   # 通用 → 不调整
}

# 自动演化的约束
AUTO_ADJUST_MAX_PER_STEP = 0.02   # 每步最多调整 ±0.02
HARD_MIN = 0.3                     # 硬下限
HARD_MAX = 0.7                     # 硬上限

# 自动调整逻辑
def _auto_adjust_zhongdao(self, feedback_score: float):
    """根据用户反馈微调权重"""
    if feedback_score > 0.7:  # 正向反馈 → 微调当前主导权重
        delta = min(AUTO_ADJUST_MAX_PER_STEP, feedback_score - 0.7)
    elif feedback_score < 0.3:  # 负向反馈 → 回调
        delta = -min(AUTO_ADJUST_MAX_PER_STEP, 0.3 - feedback_score)
    else:
        return  # 中性反馈，不调整

    new_action = clamp(self.action_weight + delta, HARD_MIN, HARD_MAX)
    new_awareness = 1.0 - new_action  # 保持总和为 1.0
    self.action_weight = new_action
    self.awareness_weight = new_awareness

    设计理由：0.5 起点体现「守中」原则 — 不预设行动或觉察哪个更重要。用户类型只做微调（±0.1），不会让初始状态偏离太远。自动演化的步进极小（±0.02），确保权重变化缓慢而稳定。

    D4：规律贡献的审核标准¶

    决策：4 类禁止内容 + 5 柱接受量表 + 需编写书面审核标准文档

    4 类禁止内容（自动审核阶段直接拒绝）：

    类别 检测方式 示例
    投机取巧 关键词匹配（走捷径/薅羊毛/套利/钻空子） "如何利用规则漏洞快速获利"
    损害他人 关键词匹配（攻击/诋毁/榨取/欺骗） "怎样让竞争对手倒闭"
    违背内求 语义判断（外部归因占比 > 0.8 + 零自省词汇） "一切都是环境的问题，只要换环境就好"
    捷径成功论 关键词+语义（速成/秘籍/不劳而获） "三天成为亿万富翁的秘诀"

    5 柱接受量表（人工审核阶段的评分标准，每柱 0-20 分，总分 ≥ 70 分通过）：

    柱名 含义 满分标准
    善行 是否引向正向行动 提供具体、可操作的向善行动路径
    善人 是否促进人的成长 关注人的内在成长而非仅外部结果
    善心 是否引导内求觉察 包含自我审视和反思的触发点
    善史 是否符合长期价值 不追求短期效果，考虑长期影响
    善修 是否可实践验证 规律可以被个人实践检验，非纯理论

    需编写文档：docs/contribution-audit-standards.md，详细记录审核流程、标准操作程序、边界案例判定指南。该文档作为规律审核员的培训材料和操作手册。
