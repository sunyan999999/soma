# SOMA v0.8.0 开发计划 — 知识图谱与推理引擎

## 设计原则

1. **连接而非重建**：不创建新的存储层，连接现有的 EpisodicStore、SemanticStore 和 WisdomEngine
2. **增量可验证**：每个子任务有明确的验收标准、测试文件和预期行为
3. **API 向后兼容**：所有变更不破坏 v0.7.0 的公开 API

---

## 架构背景（v0.7.0 现状）

当前 SOMA 的推理管道是线性的：

```
Problem → WisdomEngine.decompose() → [Focus...]
                                    ↓
              ActivationHub.activate(foci) → [ActivatedMemory...]
                                    ↓
              SOMA_Agent._answer(problem, foci, memories) → response
```

关键断点（需要 v0.8.0 修复）：
- **SemanticStore 的 NetworkX 图有 `get_neighbors()`，但检索管道从未调用它**
- **ActivationHub 名为"双向激活"，实际只是 Focus→Memory 单向**
- **没有冲突检测：两条矛盾记忆可以同时被激活且得高分**
- **没有因果推理：三元素存储存在但从不构建因果图**
- **组合模板是硬编码的，不是动态发现的**

## 任务拆分（6 个子任务 + 集成测试）

### 任务 0：语义图谱接入检索管道 ⭐ 基础

**目标**：让 SemanticStore 的图结构参与 MemoryCore 的检索过程。

**当前行为**：
- `_hybrid_search()` 只做向量检索 + 关键词检索的 RRF 融合
- SemanticStore 的图被查询但边的遍历结果被丢弃
- 一个概念的相关概念永远不会被检索到

**目标行为**：
- 当聚焦点关键词匹配到三元素图谱中的节点时，沿边遍历 1-2 跳
- 找到的邻居节点作为「扩展关键词」参与 episodic 检索
- 扩展关键词的权重按图距离衰减（1 跳 × 0.6，2 跳 × 0.36）

**涉及文件**：
| 文件 | 变更类型 | 说明 |
|------|---------|------|
| `soma/memory/semantic.py` | 修改 | 新增 `expand_keywords(node, depth=2)` 方法，沿图遍历返回邻居节点标签 |
| `soma/memory/core.py` | 修改 | `_hybrid_search()` 调用 `semantic.expand_keywords()`，将扩展词加入关键词检索 |
| `tests/test_semantic.py` | 新增测试 | 测试图遍历扩展的正确性（深度、衰减） |
| `tests/test_memory_core.py` | 新增测试 | 验证混合检索结果包含图扩展带来的相关记忆 |

**验收标准**：
- [ ] `expand_keywords("客户流失", depth=2)` 返回 `["用户留存", "产品体验", "定价策略"]` 等图邻居
- [ ] 检索 "客户流失" 时，包含 "用户留存" 节点的记忆出现在结果中（之前不会）
- [ ] 图扩展耗时 < 5ms（10K 节点规模）
- [ ] 无图扩展时检索行为不变（向后兼容）

---

### 任务 1：因果推理链

**目标**：构建因果子图，支持从现象追溯到根因的分析链。

**当前行为**：
- 三元素存储了 `(A, causes, B)` 但没有因果路径查询
- 用户问「为什么客户流失？」时，SOMA 只能返回孤立的因果陈述
- 无法展示 A→B→C→D 的因果链条

**目标行为**：
- 新增 `CausalPath` 查询：给定起始节点，在语义图谱中沿因果边（causes/leads_to/prevents/increases/decreases）遍历
- 支持「根因分析」模式：找出无前驱因果边的叶子节点
- 支持「影响分析」模式：从给定节点出发找所有下游影响

**涉及文件**：
| 文件 | 变更类型 | 说明 |
|------|---------|------|
| `soma/memory/causal.py` | **新建** | `CausalGraph` 类：因果边过滤、路径查询、根因/影响分析 |
| `soma/memory/semantic.py` | 修改 | 添加 `get_causal_edges()` 和 `get_edges_by_predicate(predicates)` |
| `soma/memory/core.py` | 修改 | 暴露 `query_causal_chain(node, direction, depth)` |
| `soma/hub/_core.py` | 修改 | 当 foci 中有 `systems_thinking` 律时，自动触发因果分析 |
| `tests/test_causal.py` | **新建** | 因果路径查询、根因发现、循环检测测试 |

**验收标准**：
- [ ] 给定 A→B→C→D 因果链，`find_root_causes("D")` 返回 `["A"]`
- [ ] 给定分叉因果，`find_all_effects("A")` 返回 A 的所有下游节点
- [ ] 因果环路 `A→B→C→A` 被检测到并标记，而非无限循环
- [ ] 空因果图返回空列表，不报错
- [ ] 因果分析耗时：1000 边图 < 10ms

### 任务 2：记忆冲突检测

**目标**：当两条被激活的记忆存在逻辑矛盾时，自动检测并标记。

**当前行为**：
- 没有冲突检测机制
- 两条矛盾的记忆可以同时被激活得高分
- `anti_confirmation_search` 只做简单的否定词匹配，不够语义化

**目标行为**：
- 对高激活记忆对进行语义矛盾检测
- 使用嵌入向量的余弦相似度 + 否定变换来判断是否矛盾
- 冲突记忆在最终排序中被降权或标记警告
- 在响应 prompt 中注入冲突提示（"以下记忆存在潜在矛盾..."）

**涉及文件**：
| 文件 | 变更类型 | 说明 |
|------|---------|------|
| `soma/hub/_conflict.py` | **新建** | `ConflictDetector` 类：嵌入向量矛盾检测、冲突评分、证据强度计算 |
| `soma/hub/_ranker.py` | 修改 | 在 MMR 之后加入冲突降权步骤 |
| `soma/hub/_core.py` | 修改 | 集成 `ConflictDetector`，在 activate 返回前运行 |
| `soma/agent.py` | 修改 | 在 prompt 中注入冲突提示信息 |
| `tests/test_conflict.py` | **新建** | 矛盾检测准确性、误报率、性能测试 |

**验收标准**：
- [ ] «价格是客户流失的主因» vs «价格不是问题，服务质量才是» → 检测为矛盾（相似度 > 阈值且语义极性相反）
- [ ] «天气影响销量» vs «促销活动提升销量» → 不检测为矛盾（可以同时成立）
- [ ] 冲突检测不对性能造成显著影响（< 20ms 额外延迟）
- [ ] 误报率 < 15%（在人工标注的测试集上）

### 任务 3：双向激活闭环

**目标**：让高激活记忆能够「反馈」到思维框架，触发额外的思维规律或聚焦点。

**当前行为**：
- Focus → Memory：思维规律激活记忆（单向）
- 名称叫 "bi-directional" 但实际只有一个方向

**目标行为**：
- 被激活的记忆如果有 `suggested_law` 元数据 → 触发对应思维规律
- 被激活记忆的领域标签（domain）映射到思维规律 → 提升相关规律权重
- 在 `ActivationHub.activate()` 返回的 `ActivatedMemory` 中增加 `suggested_focus` 字段
- WisdomEngine 在下一次分解时考虑记忆建议的修正

**涉及文件**：
| 文件 | 变更类型 | 说明 |
|------|---------|------|
| `soma/hub/_core.py` | 修改 | activate() 后运行 memory→focus 反向传播 |
| `soma/hub/_scorer.py` | 修改 | 计算 memory.suggested_law 的激活潜力 |
| `soma/engine.py` | 修改 | decompose() 接受可选的 `memory_suggestions` 参数 |
| `soma/base.py` | 修改 | ActivatedMemory 新增 `suggested_focus: Optional[Focus]` 字段 |
| `tests/test_hub.py` | 修改 | 测试反向激活的正确性 |

**验收标准**：
- [ ] 一条带有 `suggested_law: "systems_thinking"` 的记忆被激活后，下一次 decompose 中 systems_thinking 权重提升
- [ ] 反向激活的影响不超过直接触发的 50%（避免记忆过度影响思维框架）
- [ ] 无 `suggested_law` 的记忆不产生虚假的反向激活
- [ ] 反向激活逻辑在 `explain_activation()` 中可追溯

### 任务 4：跨领域类比引擎

**目标**：当问题域记忆中内容较少时，从其他领域寻找结构相似的记忆模式。

**当前行为**：
- 检索只在内容相关的记忆中搜索
- 如果问题域（如「医疗」）记忆中信息少，检索结果稀疏
- 没有「这个医疗问题本质上和机械工程中的故障诊断是同一个结构」的能力

**目标行为**：
- 提取当前问题的「结构特征」——不是内容关键词，而是关系模式
- 在语义图谱中搜索具有相似关系结构的其他领域
- 将找到的类比领域的记忆作为「补充视角」加入结果
- 标注为类比结果（不同于直接检索结果）

**结构特征提取**：
```
问题: "如何降低客户流失率？"
结构: [实体: 客户] --[减少]--> [指标: 流失率]
类比: [实体: 热力学系统] --[减少]--> [指标: 熵增]
      → 从热力学中借用"封闭系统熵增不可避免"的洞察
```

**涉及文件**：
| 文件 | 变更类型 | 说明 |
|------|---------|------|
| `soma/engine/analogy.py` | **新建** | `AnalogyEngine`：结构模式提取 + 跨域图谱匹配 |
| `soma/hub/_retriever.py` | 修改 | 当某领域记忆 < 阈值时，触发类比检索 |
| `soma/hub/_core.py` | 修改 | 在结果中标记类比来源 |
| `soma/base.py` | 修改 | ActivatedMemory 新增 `source_type: str` 字段（direct/analogy） |
| `tests/test_analogy.py` | **新建** | 结构匹配准确性、跨域检索测试 |

**验收标准**：
- [ ] 问题域记忆 < 3 条时自动触发类比检索
- [ ] 返回的类比记忆与原问题有结构相似性（同构子图匹配得分 > 0.5）
- [ ] 类比结果标注为 `source_type: "analogy"`，与直接检索区分
- [ ] 类比检索不增加超过 50ms 的延迟

### 任务 5：反思质量自评

**目标**：SOMA 对自己的回答进行质量评估，低分回答自动重新生成。

**当前行为**：
- LLM 生成的反思直接存储
- 没有质量评估机制
- 低质量反思和高质量反思权重相同地参与演化

**目标行为**：
- 每次 `respond()` 后运行质量自评
- 评估维度：内部一致性（引用记忆是否支持结论）、逻辑连贯性、可操作性
- 低于阈值的回答自动重新生成（最多 3 次）
- 质量分数影响记忆存储时的 importance 权重

**评分机制**：
```
quality_score = 0.4 * consistency_score    # 记忆是否支撑结论
              + 0.3 * coherence_score       # 逻辑是否连贯
              + 0.3 * actionability_score   # 是否给出可执行建议
```

**一致性评分**（不需要 LLM，本地计算）：
- 被引用的记忆嵌入向量 vs 结论嵌入向量的余弦相似度
- 如果结论远离所有引用记忆 → 一致性低

**连贯性评分**（需要轻量 LLM 调用，或本地规则）：
- 段落间是否存在逻辑跳跃
- 是否有「因为 A 所以 B」的因果连接词

**可操作性评分**（本地规则）：
- 是否包含具体的步骤、数字、时间线
- 关键词匹配：百分比、天/周/月、步骤 1/2/3

**涉及文件**：
| 文件 | 变更类型 | 说明 |
|------|---------|------|
| `soma/quality.py` | **新建** | `QualityEvaluator`：三维度评分、阈值判定、重试逻辑 |
| `soma/agent.py` | 修改 | respond() 后调用 QualityEvaluator，低分触发重生成 |
| `soma/evolve.py` | 修改 | reflect() 中考虑质量分数 |
| `soma/config.py` | 修改 | 新增 `quality_threshold` 配置项（默认 0.7） |
| `tests/test_quality.py` | **新建** | 评分准确性、重试机制、配置生效测试 |

**验收标准**：
- [ ] 高质量回答（引用了相关记忆、逻辑连贯、可操作）→ 得分 > 0.7
- [ ] 低质量回答（未引用记忆、逻辑跳跃、空洞）→ 得分 < 0.5
- [ ] 得分 < 0.7 时自动重试，最多 3 次，取最高分
- [ ] 3 次重试后仍低分 → 返回最佳结果但标记为「低置信度」
- [ ] 自评耗时 < 200ms（不含 LLM 调用时）或 < 2s（含轻量 LLM 时）

## 执行顺序

```
任务 0 (语义图谱接入) ──→ 任务 1 (因果推理链)
     ↓                          ↓
任务 2 (冲突检测)    ←── 任务 0 + 任务 1 的图谱数据
     ↓
任务 3 (双向激活)    ←── 依赖任务 2 的冲突信息
     ↓
任务 4 (跨域类比)    ←── 依赖任务 0 的图谱扩展 + 任务 3 的激活数据
     ↓
任务 5 (反思质量)    ←── 独立，可与前面并行
     ↓
任务 6 (集成测试)    ←── 全部完成后
```

## 关键设计决策

1. **不引入新依赖**：因果图、类比引擎均基于已有的 NetworkX + NumPy，不新增 PyPI 依赖
2. **图谱查询惰性化**：只在需要时才做图遍历，不影响普通的快速检索路径
3. **冲突检测本地化**：使用嵌入向量相似度计算，不调用 LLM（避免额外延迟和成本）
4. **质量评估分阶段**：一致性评估用本地嵌入，连贯性评估可选 LLM，可操作性评估纯规则
5. **向后兼容**：所有新参数都有默认值，关闭新功能后行为与 v0.7.0 完全一致
