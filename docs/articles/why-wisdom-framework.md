## 一、RAG 的尽头是一个更好的搜索引擎吗？

过去两年，RAG（检索增强生成）成了 AI 应用的标配。几乎所有 AI Agent 架构都是：

```
用户问题 → 向量检索 → 拼接上下文 → LLM 生成答案
```

这套模式在问答场景下效果很好。但当你试图构建一个**真正能持续成长的 AI 伙伴**时，问题就来了：

1. **知识碎片化**：200 条相关片段拼接在一起，LLM 看到的是一堆信息碎片，而非结构化思维
2. **记忆僵硬**：所有记忆权重相等，无法区分"一周前的 bug"和"一年前的人生洞察"
3. **不会反思**：纯检索系统不知道自己检索得对不对，更不会修正自己的检索策略

**SOMA 的做法是**：在检索和生成之间，插入一个**显式的思维框架层**。

## 二、7 条思维规律驱动的记忆激活

SOMA 的核心是一个可拔插的思维框架，由 7 条哲学规律构成：

| 规律 | 核心问题 | 权重 |
|------|---------|:--:|
| 第一性原理 | 事物最基本的要素是什么？ | 0.95 |
| 系统思维 | 要素之间的连接和反馈回路？ | 0.90 |
| 矛盾分析 | 表面下的深层对立统一关系？ | 0.85 |
| 二八法则 | 哪 20% 的关键因素决定 80% 的结果？ | 0.80 |
| 逆向思考 | 反过来想，会怎样？ | 0.75 |
| 类比推理 | 与什么相似？可以借鉴什么？ | 0.70 |
| 进化视角 | 这件事如何随时间演变？ | 0.65 |

当你问 SOMA "为什么增长停滞？"时，它不会直接搜索"增长停滞"相关记忆。而是：

```
问题: "为什么增长停滞？"

拆解 (Decompose):
  ├── 焦点 1: 回归产品价值的基本要素 [第一性原理]
  ├── 焦点 2: 分析各要素的负反馈回路 [系统思维]  
  ├── 焦点 3: 找出深层对立的矛盾 [矛盾分析]
  └── 焦点 4: 识别 20% 的核心瓶颈 [二八法则]

激活 (Activate):
  ├── 记忆 #42: "第一性原理在商业中意味着关注客户最根本的需求" (0.87)
  ├── 记忆 #17: "增长停滞是多要素形成负反馈回路的结果" (0.91)
  └── 记忆 #55: "表面问题是获客不足，深层矛盾是价值交付差距" (0.84)

合成 (Synthesize):
  → 基于框架指令 + 激活记忆 → 构建结构化 Prompt → LLM 生成答案

进化 (Evolve):
  → 记录本次各规律的激活次数 → 每10次会话自动微调权重(±2%)
```

这不是 prompt engineering，而是**架构层的设计决策**。

## 三、技术栈：零依赖的极致轻量

SOMA 的哲学是"单机优先"：

```
SQLite (FTS5 trigram 全文索引)  →  中文关键词搜索 <10ms @100K记忆
FAISS (IVF 向量索引)            →  语义相似搜索
ONNX (fastembed CPU 推理)       →  嵌入向量生成，无需 GPU
Vue 3 + ECharts                →  仪表盘可视化
```

不需要 Docker，不需要 Redis，不需要 GPU。一个 `pip install soma-wisdom` 就能跑。

**v0.3.1b1 性能基准**：

| 指标 | 1K 记忆 | 10K 记忆 | 100K 记忆 |
|------|:--:|:--:|:--:|
| 关键词搜索 | <3ms | <5ms | <10ms |
| 语义搜索 | ~10ms | ~30ms | ~100ms |
| 记忆写入 | ~3ms | ~5ms | ~8ms |
| FTS5 加速比 | — | 25x vs LIKE | 50x+ vs LIKE |

## 四、系统会自我进化

SOMA 不只是用思维框架检索记忆，它还会**反思和改进这个框架本身**。

**自主规律发现**（v0.3.0）：
1. DBSCAN 聚类识别记忆中的高频共现模式
2. TF-IDF 提取候选关键词
3. LLM 审核（或启发式兜底）：生成新规律的 name/description/triggers
4. 人工确认后纳入框架
5. 安全熔断：单次最多 1 条，总数上限 15 条

**元认知进化**（v0.2.0）：
- 每 10 次会话触发一次进化
- 分析各规律的激活频率和成功率
- 激活多且成功率高的规律 → +2% 权重
- 激活少且成功率低的规律 → -2% 权重

| **自我进化** | ❌ | ❌ | ❌ | ✅ 权重新规律 |
