## 总体判断

### 可行性：全部可落地

六个模块均可在 SOMA v0.9.0 基础上以**增量方式**实现，无需重写核心架构。每个模块都有明确的现有代码锚点作为扩展基础。

### 核心洞察

当前 SOMA 架构存在一条清晰的「认知处理链」：

```
问题进入 → 拆解 → 记忆激活 → 冲突检测 → 推理框架 → LLM调用 → 质量评估 → 反思记录
```

六个模块分别在这条链的不同位置接入：

| 模块 | 接入位置 | 侵入性 |
|------|---------|--------|
| 内求复盘钩子 | 管道每步之后 | 低（钩子注入） |
| 守中执一算法 | 冲突检测 / 共识协议 | 低（新增策略） |
| 认知熵值仪表盘 | 管道末尾（观察者） | 零（独立模块） |
| 空性偏差检测 | 冲突检测阶段 | 极低（追加函数调用） |
| 商道罗盘/企熵诊断仪 | 管道入口（路由前） | 中（需插件系统） |
| 规律贡献接口 | 规律发现 + 引擎层 | 中（审核流程+存储） |

### 推荐实施顺序

```
第一阶段（1-2周）         第二阶段（2-3周）        第三阶段（3-4周）
┌──────────────┐       ┌──────────────┐       ┌──────────────┐
│ 认知熵值仪表盘 │  →   │ 空性偏差检测   │  →   │ 内求复盘钩子   │
│ (独立，快速验证) │      │ (小改动，高效果) │      │ (钩子框架先行) │
└──────────────┘       └──────────────┘       └──────────────┘

第四阶段（3-4周）        第五阶段（4-6周）        第六阶段（6-8周）
┌──────────────┐       ┌──────────────┐       ┌──────────────┐
│ 守中执一算法   │  →   │ 商道罗盘      │  →   │ 规律贡献接口   │
│ (≤50行核心函数) │      │ (需插件系统就绪) │      │ (最大范围)    │
└──────────────┘       └──────────────┘       └──────────────┘
```

## 模块一：内求式复盘钩子

### 能力描述

在 SOMA 的十步推理管道中，为每个步骤植入可配置的「觉察触发器」。不是事后的定期复盘，而是推理进行中的实时内省提示。

### 现有基础

| 资产 | 位置 | 用途 |
|------|------|------|
| `soma.plugin.soma.hooks` 入口点组 | [plugin.py](soma/plugin.py#L19) | 已定义但未接入管道 |
| `SOMA_Agent.respond()` 管道步骤 | [agent.py](soma/agent.py#L186-274) | 8 个明确步骤，每步后都是钩子注入点 |
| `MetaEvolver.reflect()` | [evolve.py](soma/evolve.py#L167-215) | 现有事后复盘机制，可复用其统计上下文 |
| `_assess_complexity()` | [agent.py](soma/agent.py#L277-290) | complexity_score 直接可用作触发条件 |
| `QualityEvaluator` | [quality.py](soma/quality.py#L196-231) | 一致性/连贯性评估可作钩子输入 |

### 内置钩子设计（第一期 6 个）

| 钩子 | 触发步 | 触发条件 | 提示示例 |
|------|--------|---------|---------|
| 复杂度觉察 | decompose 后 | complexity_score > 6 | "这个问题复杂度较高，你第一反应是用哪种经典框架去分析？这个框架是否完全贴合当前场景？" |
| 情绪负载觉察 | activate 后 | emotional_load > 0.7 | "我注意到您可能带着较强的情绪参与此次思考。建议先暂停几秒，感受一下呼吸，再决定是否采纳此建议。" |
| 框架惯性觉察 | reasoning 后 | 连续 3 轮使用同一框架 | "您已连续多次使用{framework}框架分析问题，是否可以考虑切换到另一个维度？" |
| 信息缺失觉察 | prompt_build 前 | 关键信息缺失 flag | "在给出结论前，我注意到还缺少关于{missing_info}的信息，是否先补充后再继续？" |
| 行动外求觉察 | action 后 | 建议中外部依赖占比 > 0.7 | "这个行动建议较多依赖外部因素，在等待外部变化的同时，您自己可以先做的一小步是什么？" |
| 确定性觉察 | respond 结束 | 建议确定性 > 0.9 | "我对这个结论的确定性较高，但仍想邀请您用自己的判断再审视一下——这与您的直觉一致吗？" |

### 复杂度评估

- **新增文件**: 4 个（~350 行）
- **修改文件**: 1 个（agent.py，约 +30 行）
- **测试**: ~10 项（钩子注册/触发/配置热加载/管道注入）
- **风险**: 低。钩子失败不应阻塞主管道，所有钩子调用包在 try/except 中
- **工期**: 约 1 周

```python
# soma/multi_agent/zhongdao.py

def compute_zhongdao_score(
    candidate_action: str,
    user_profile: dict,
    conversation_context: dict,
    *,
    action_weight: float = 0.5,    # 行动力权重（可配置）
    awareness_weight: float = 0.5,  # 觉知力权重（可配置）
) -> float:
    """
    计算候选行动方案的「中道」得分 (0~1)。
    得分越高，越接近「向外解决」与「向内回看」的平衡。
    """
    score = 0.0
    
    # 维度 1: 内在熵减评估 (权重 0.3)
    # 分析该行动是否会减少用户的内在混乱度
    entropy_delta = _estimate_entropy_change(user_profile, candidate_action)
    score += 0.3 * (1.0 - entropy_delta)  # 熵减得分高
    
    # 维度 2: 外部依赖评估 (权重 0.4)
    # 行动建议中外部因素占比越高，越偏离「内求」
    external_ratio = _extract_external_dependency(candidate_action)
    score += 0.4 * (1.0 - external_ratio)
    
    # 维度 3: 长期成长对齐 (权重 0.3)
    # 与用户历史成功案例的语义相似度
    growth_alignment = _compute_growth_alignment(
        candidate_action, user_profile.get("success_history", [])
    )
    score += 0.3 * growth_alignment
    
    return min(max(score, 0.0), 1.0)

## 模块四：空性偏差检测升级

### 能力描述

在现有偏差检测基础上，新增「框架锚定识别」——检测用户是否过度锁定单一分析框架（如总从外部归因），并温和提示切换到其他视角。这体现了「空性」智慧：任何判断都依赖于特定框架，看见框架本身就是拓展认知边界。

### 现有基础

| 资产 | 位置 | 用途 |
|------|------|------|
| `ConflictDetector` | [hub/_conflict.py](soma/hub/_conflict.py) | 现有的记忆冲突检测，扩展点清晰 |
| `ActivationHub.anti_confirmation_search()` | [hub/_core.py](soma/hub/_core.py#L135-177) | 已有反视角检索，新的偏差类型可在此追加 |
| `ActivationHub.activate()` | [hub/_core.py](soma/hub/_core.py#L40-106) | 偏差检测结果从这传入 prompt |
| `_build_prompt()` | [agent.py](soma/agent.py#L401-520) | 已有「反面视角与潜在矛盾」section，可扩展 |

### 需要新建

```

### 核心类设计

```python
# soma/hub/_frame_detector.py

class FrameAnchoringDetector:
    """检测用户是否过度锁定单一分析框架"""
    
    def __init__(self, keywords_config: Path = None):
        self.frames = self._load_frames(keywords_config)
        self.window_size = 5  # 最近 N 轮对话
    
    def detect(self, conversation_turns: List[dict]) -> Optional[dict]:
        """
        返回:
        None — 无框架锚定
        {
            "frame": "外部归因",
            "ratio": 0.75,
            "opposite_frame": "内在归因",
            "reflection": "我注意到您在过去几次对话中，有75%的分析都采用了「外部归因」框架..."
        }

class PatternSubmission:
    """用户提交的规律"""
    pattern_name: str
    description: str
    category: str
    contributor: str
    trigger_conditions: List[dict]
    processing_flow: List[dict]
    effect_metrics: List[str]
    empirical_basis: str
    
    # 审核状态
    status: str = "pending"  # pending / auto_reviewed / approved / rejected
    energy_score: int = 0     # 被采纳能量值
    embedding: Optional[np.ndarray] = None


class ContributionHandler:
    """规律贡献处理器"""
    
    def __init__(self, engine: WisdomEngine, embedder, store: ContributionStore):
        self.engine = engine
        self.embedder = embedder
        self.store = store
    
    def submit(self, submission: PatternSubmission) -> str:
        """提交新规律，返回 submission_id"""
        # 1. 格式校验 (JSON Schema)
        # 2. 计算 embedding
        # 3. 与现有规律做去重检查（相似度 < 0.85 才通过）
        # 4. 持久化到 contribution_store
        # 5. 返回 submission_id 供跟踪
    
    def auto_review(self, submission_id: str) -> dict:
        """自动第一轮审核"""
        # 格式完整性 + 与已有规律去重 + 触发条件合理性
    
    def manual_approve(self, submission_id: str, reviewer_id: str) -> bool:
        """人工审核通过，纳入规律网络"""
        # 生成 pattern_id，加入 wisdom_laws.yaml 和向量索引
    
    def record_adoption(self, pattern_id: str, outcome: str):
        """记录规律被采纳的结果"""
        # 成功 → energy_score +1
        # 负面反馈 → energy_score -2
        # 低于阈值 → 自动降级，不再参与检索
    
    def get_active_patterns(self, context: dict, limit: int = 5) -> list:
        """检索当前上下文应激活的贡献规律"""
        # 基于语义匹配 + energy_score 阈值过滤
```

### 待讨论决策问题

以下问题建议与文化赋能师和开发者一起讨论后确定：

**D1 — 钩子的侵入程度**：钩子提示是仅记录在日志中（静默），还是实时呈现给用户？如果实时呈现，以什么形式（附加在回答末尾 / 单独提示 / 仅对开发者可见）？

**D2 — 认知熵值的「觉察引导模式」触发阈值**：连续几轮熵值低于多少时切换？50 分？40 分？切换后如何恢复？

**D3 — 中道算法的「阴阳权重」初始值**：行动力(阳)和觉知力(阴)各 0.5 是否合理？不同用户类型是否需要不同初始配置？

**D4 — 规律贡献的价值观边界**：什么算「善」的规律？是否需要一份明确的价值观审核标准文档？

**D5 — 贡献规律的版本管理**：当原始贡献者修改一个已被广泛采纳的规律时，如何处理——更新还是分叉？

**D6 — 零熵测试系统的集成程度**：认知熵值和零熵现有的「能量信息测试系统」之间是什么关系——独立并行、数据互相参照、还是深度集成？

## 总结

六个模块的整合不是向 SOMA「添加功能」，而是为其注入一种**全新的认知范式**。当前 SOMA 是一个高效的问题解决引擎——它能思考、能记忆、能进化。整合零熵智慧后，SOMA 将增加一个新的维度：**觉察**。

它不只帮用户「想得更清楚」，还能帮用户「看见自己在怎么想」。

技术上，六个模块的设计遵循了同一个约束：**每个模块都可以独立开关、独立迭代、独立验证**。没有一个是必须依赖其他模块才能运行的。这让实施可以灵活推进，有任何模块达不到预期都可以单独回退而不影响整体。
