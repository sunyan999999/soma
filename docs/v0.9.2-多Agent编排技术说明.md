# SOMA v0.9.2 多Agent编排层技术说明

> 写给零熵智库开发者（Qoder）的对接文档。描述编排层架构、数据流、接口，以及两套系统组合的最佳切入点。

---

## 一、架构总览

```
SOMA 门面 (soma/__init__.py)
  │
  ├─ orchestration_mode="single" → 直接调用 SOMA_Agent.respond()
  │
  └─ orchestration_mode="multi"  → SOMAOrchestrator.solve()
        │
        ├─ 1. AgentRegistry   — 管理已注册的专家Agent
        ├─ 2. ExpertRouter    — 按问题关键词/语义匹配专家
        ├─ 3. Dispatch        — 向匹配到的专家逐个提问
        └─ 4. ConsensusProtocol — 综合多专家意见形成共识
```

零修改侵入：编排层在 SOMA 门面层完成路由切换，不修改 SOMA_Agent 核心管道。

## 二、核心组件

### 2.1 AgentRegistry（注册表）

管理所有专家 Agent 的生命周期。每个专家包含：

| 字段 | 类型 | 说明 |
|------|------|------|
| `agent_id` | str | 唯一标识 |
| `expertise` | list[str] | 专长领域标签，如 `["商业分析", "战略"]` |
| `description` | str | 人类可读描述 |
| `session_count` | int | 累计参与会话数 |
| `success_rate` | float | 成功率（指数移动平均） |

关键方法：

```python
registry.register(agent, expertise, description, is_default=False) -> str
registry.unregister(agent_id) -> bool
registry.find_experts(domain, min_score=0.3) -> List[Tuple[agent, float]]
registry.get(agent_id) -> agent | None
registry.list_agents() -> List[AgentInfo]
```

**对接要点**：`register()` 接受的 `agent` 可以是任何实现了 `respond(problem, user_id="") -> str` 方法的对象。不需要是 SOMA_Agent 的子类。

### 2.2 ExpertRouter（路由器）

三级路由策略：

| 级别 | 策略 | 说明 |
|------|------|------|
| L1 | 关键词匹配 | 问题分词后匹配 expertise 关键词映射表 |
| L2 | 语义相似度 | 问题 embedding 与专家描述的余弦相似度 |
| L3 | 回退 | 无匹配时使用默认 Agent |

L1 关键词映射表（当前硬编码，可扩展）：

```python
{
    "技术":   ["代码", "架构", "API", "数据库", "部署", "算法", "性能", "bug", "编程"],
    "商业分析": ["商业", "市场", "营销", "增长", "竞品", "营收", "定价", "客户"],
    "设计":   ["UI", "UX", "交互", "视觉", "体验"],
    "管理":   ["团队", "组织", "领导", "绩效", "流程"],
    # ... 可通过 registry.register() 时指定的 expertise 动态扩展
}
```

路由返回：`List[(agent, score, strategy)]`，其中 strategy ∈ `{"l1", "l2", "fallback"}`

### 2.3 ConsensusProtocol（共识协议）

三级共识策略：

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| `voting` | 加权多数投票（confidence × 记忆支持数） | 默认，快速决策 |
| `llm_arbitration` | LLM 仲裁分歧 | 中等分歧 |
| `dialectical_synthesis` | 辩证综合（正反合） | 深层哲学问题 |

输出 `ConsensusResult`：

```python
@dataclass
class ConsensusResult:
    consensus_answer: str       # 最终共识回答
    agreement_level: float      # 共识度 0-1
    agent_opinions: list        # 各专家原始观点
    strategy_used: str          # 使用的策略
    disagreements: list         # 分歧点
    minority_view: str | None   # 少数派观点
```

单专家场景：直接返回该专家观点，agreement_level=1.0，无额外共识处理。

### 2.4 SOMAOrchestrator（编排器）

串联上述三者的总控：

```python
class SOMAOrchestrator:
    def create_agents(specs) -> list[str]   # 批量创建+注册专家
    def set_default(agent) -> None          # 设置默认回退Agent
    def remove_agent(agent_id) -> bool      # 移除专家
    def solve(problem, strategy, top_k) -> OrchestrationResult  # 核心管道
```

## 三、solve() 管道全流程

```
用户问题: "如何优化数据库查询性能？"
         │
         ▼
┌─ 1. Route ─────────────────────────────────────────┐
│  ExpertRouter.route_multi(problem, top_k=3)        │
│                                                     │
│  L1: "数据库"+"性能" → 命中"技术"标签                │
│  → 返回: [(tech_agent, 0.85, "l1")]                 │
│                                                     │
│  如L1无匹配 → L2语义 → 仍无匹配 → L3回退默认Agent    │
└────────────────┬────────────────────────────────────┘
                 │
                 ▼
┌─ 2. Dispatch ──────────────────────────────────────┐
│  for agent, score, strategy in experts:             │
│      try:                                           │
│          answer = agent.respond(problem)             │
│          opinions.append(AgentOpinion(...))          │
│      except Exception:                              │
│          记录失败，继续下一个专家                     │
│                                                     │
│  关键：单个专家异常不影响其他专家（G1错误隔离）        │
│  如所有专家失败 → 返回空共识，门面层回退单Agent       │
└────────────────┬────────────────────────────────────┘
                 │
                 ▼
┌─ 3. Consensus ─────────────────────────────────────┐
│  ConsensusProtocol.form_consensus(                  │
│      problem, opinions, strategy="voting"           │
│  )                                                  │
│                                                     │
│  1个专家 → 直接返回，agreement_level=1.0             │
│  2+专家 → 加权投票 / LLM仲裁 / 辩证综合              │
│                                                     │
│  输出：ConsensusResult（含最终回答、共识度、分歧点）   │
└────────────────┬────────────────────────────────────┘
                 │
                 ▼
┌─ 4. Return ────────────────────────────────────────┐
│  OrchestrationResult {                              │
│      answer: str,              # 最终回答            │
│      agents_involved: [str],   # 参与专家列表         │
│      routing_strategy: str,    # "l1"/"l2"/"fallback"│
│      consensus: ConsensusResult | None               │
│  }                                                  │
└─────────────────────────────────────────────────────┘
```

错误处理：solve() 内部任何异常返回 `OrchestrationResult(answer="多Agent编排过程出错: ...")`，上层 SOMA.chat() 检测 consensus 为 None 时自动回退到单Agent管道。

## 四、REST API

所有端点已实装在 `dash/server.py`：

| 端点 | 方法 | 说明 |
|------|------|------|
| `/api/experts` | GET | 列出所有已注册专家 |
| `/api/experts` | POST | 注册新专家 |
| `/api/experts/{id}` | DELETE | 移除专家 |
| `/api/orchestration/status` | GET | 编排器状态（agent数/路由统计/共识会话数） |
| `/api/chat` | POST | 对话接口（自动走编排管道） |
| `/api/chat/stream` | POST | 流式对话（同上） |

注册专家请求体：
```json
{
    "agent_id": "tech_expert",
    "expertise": ["技术", "架构"],
    "description": "技术架构专家"
}
```

## 五、与零熵智库组合方案

### 当前状态

两套系统各有独立的路由：

```
零熵智库:  twin 参数 → 指定分身 → 分身知识库 → 回答
SOMA:      problem 关键词 → 匹配专家 → 专家.respond() → 共识
```

### 推荐组合方式

让 SOMA 的专家不是"空白 SOMA_Agent"，而是"零熵智库分身的适配器"：

```python
class TwinExpertAdapter:
    """包装零熵智库分身，使其可作为 SOMA 专家参与编排"""
    
    def __init__(self, twin_id: str, twin_api_url: str):
        self.agent_id = twin_id
        self.twin_api_url = twin_api_url
    
    def respond(self, problem: str, user_id: str = "") -> str:
        """委托给零熵智库分身回答"""
        # POST {twin_api_url}/chat  {"problem": problem}
        # 返回分身的回答
        response = requests.post(
            f"{self.twin_api_url}/chat",
            json={"problem": problem, "user_id": user_id},
        )
        return response.json()["answer"]
```

然后在 SOMA 启动时将零熵智库的分身列表同步注册：

```python
# SOMA 启动后
for twin in ling_shang_twins:
    adapter = TwinExpertAdapter(
        twin_id=twin["id"],
        twin_api_url=f"https://智库API/twins/{twin['id']}",
    )
    soma._orchestrator.registry.register(
        adapter,
        expertise=twin["expertise"],   # 从智库分身配置获取
        description=twin["description"],
    )
```

### 组合后的请求流程

```
用户: "如何做竞品分析？"
        │
        ▼
SOMA Orchestrator
  ├─ Router: "竞品"关键词 → 匹配 expertise=["商业分析"] → 孙岩分身
  ├─ Dispatch: TwinExpertAdapter.respond() → 调用零熵智库孙岩分身API
  ├─ 孙岩分身: 用自己的专业知识库 + SOMA思维框架 → 深度分析
  └─ Consensus: 若还匹配了其他分身 → 综合各方观点
```

### 需要零熵智库提供

1. **分身查询接口**：获取所有分身的 `id`、`expertise`（专长标签）、`description`
2. **分身对话接口**：`POST /chat`，接受 `{"problem": str}`，返回 `{"answer": str}`
3. （可选）**流式对话接口**：SSE 格式

### SOMA 侧需要做的

1. 实现 `TwinExpertAdapter`（~30行）
2. 启动时自动同步分身列表到 Registry（~20行）
3. 零改动侵入现有代码

## 六、当前限制

1. **Dispatch 串行执行**：v0.9.2 逐个调用专家，并行留后续版本
2. **专家不具备独立记忆**：Adapter 模式下零熵智库分身自带知识库，无此限制
3. **共识策略有限**：voting 不依赖 LLM，llm_arbitration 和 dialectical_synthesis 需额外 LLM 调用
